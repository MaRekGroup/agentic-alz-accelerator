What This Accelerator Does

End-to-end automation from requirements to continuous monitoring.

🚀

Deploy

Conversational agents gather requirements, run adversarial review, and deploy Bicep or Terraform IaC with what-if preview and human approval at every gate.

🔭

Monitor

Continuous compliance scans every 30 minutes, hourly drift detection via Resource Graph change tracking, and real-time security posture from Defender for Cloud.

🔧

Remediate

8 built-in auto-remediation strategies with pre-action snapshots for rollback. Critical and high violations auto-fixed; medium/low escalated for human approval.

Key Design Principles

Built on APEX patterns and Azure Cloud Adoption Framework guidance.

🏛️ CAF-Aligned

Every IaC module and agent maps to one of the 8 official Azure Landing Zone design areas: Billing & Tenant, Identity, Resource Org, Networking, Security, Management, Governance, Platform Automation.

⚔️ Adversarial Review

The Challenger agent reviews outputs at gates 1, 2, 4, and 5 with depth proportional to complexity: 1× for Simple, 2× for Standard, 3× for Complex deployments.

🛡️ Security-First

6 non-negotiable baseline rules + 7 extended anti-pattern checks enforced at code generation, deploy preflight, pre-commit hooks, CI, and continuous monitoring.

💰 Cost Governance Built-In

Budget alerts at 80/100/120% forecast are mandatory in every deployment. "No budget, no merge" — enforced by validators at commit time and in CI.

🔌 MCP Server Integration

3 MCP servers connect agents to live Azure data: Azure Pricing (cost estimation), Azure Platform (consolidated Resource Graph, Policy, Deployment & Monitor tools), and Draw.io (architecture diagrams). Real-time cost estimates, drift detection, compliance queries, and diagram generation.

🤝 Human-in-the-Loop

6 approval gates ensure humans decide at every critical juncture. Agents orchestrate and prepare — humans validate before anything runs against production Azure.

Installation

Get the accelerator running in under 10 minutes.

Step 1 — Clone and set up environment

# Clone the repository git clone https://github.com/your-org/agentic-alz-accelerator.git cd agentic-alz-accelerator # Create virtual environment python -m venv .venv source .venv/bin/activate # Linux/Mac # .venv\Scripts\activate # Windows # Install Python dependencies pip install -r requirements.txt

Step 2 — Configure environment variables

# Copy the example config cp .env.example .env # Edit with your values AZURE_AI_PROJECT_CONNECTION_STRING=your-connection-string AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/ AZURE_OPENAI_DEPLOYMENT=gpt-4o AZURE_SUBSCRIPTION_ID=your-subscription-id IAC_FRAMEWORK=bicep # or terraform

Step 3 — Authenticate to Azure

# Interactive login (recommended for dev) az login # Or use a service principal (recommended for CI/CD) az login --service-principal \ --username $AZURE_CLIENT_ID \ --password $AZURE_CLIENT_SECRET \ --tenant $AZURE_TENANT_ID

Step 4 — Install pre-commit hooks (optional but recommended)

# Install Lefthook npm install # Install the hooks into git npx lefthook install # Verify hooks are installed npx lefthook run pre-commit --all-files

Step 5 — Run your first workflow

# Full APEX workflow with approval gates (recommended) python -m src.agents.orchestrator \ --mode workflow \ --project my-first-landing-zone \ --iac-tool bicep # Or deploy directly with a profile python -m src.agents.orchestrator \ --mode deploy \ --profile corp # Continuous monitoring only python -m src.agents.orchestrator --mode monitor

Security Baseline

Non-negotiable rules enforced at every stage. Zero tolerance — violations block deployment.

🔴 Blocking Violations

Any of the 6 baseline rules missing from your IaC code will block the pipeline. The validator returns a non-zero exit code that fails the CI job.

⚠️ Extended Anti-Patterns

7 extended checks are warnings by default but escalate to blocking in production environments. Configured via agent_config.yaml.

1

TLS 1.2 Minimum

All storage accounts, SQL servers, and App Services must enforce TLS 1.2 or higher. No negotiation to lower versions.

minimumTlsVersion: 'TLS1_2' min_tls_version = "1.2"
2

HTTPS-Only Traffic

Storage accounts and web apps must reject HTTP. No mixed-mode traffic.

supportsHttpsTrafficOnly: true https_traffic_only_enabled = true
3

No Public Blob Access

Public blob access must be explicitly disabled on all storage accounts. Containers cannot be set to anonymous read.

allowBlobPublicAccess: false allow_nested_items_to_be_public = false
4

Managed Identity Preferred

Resources must use System-Assigned Managed Identity wherever supported. Connection strings with credentials are not allowed.

identity: { type: 'SystemAssigned' } identity { type = "SystemAssigned" }
5

Azure AD-Only SQL Authentication

SQL servers must disable local SQL authentication. Azure AD is the only allowed authentication method.

azureADOnlyAuthentication: true azuread_authentication_only = true
6

Public Network Disabled (Prod)

Production deployments must disable public network access on all PaaS services. Traffic routes through private endpoints.

publicNetworkAccess: 'Disabled' public_network_access_enabled = false

Alert Thresholds

🔴 Critical

Immediately routed to Mender for auto-remediation. Teams + email alert sent.

🟠 High

15-minute grace period then routed to Mender. Teams alert sent.

🟡 Medium

Aggregated in daily compliance report. Requires human approval for remediation.

🟢 Low

Logged in compliance report only. Reviewed in weekly platform review.

Run Monitoring

# Run compliance scan across all 4 platform subscriptions gh workflow run "monitor.yml" -f scan_type=compliance -f scan_scope=all # Scan a single landing zone gh workflow run "monitor.yml" -f scan_type=compliance -f scan_scope=security # View scan results gh run view <run-id> --log 2>/dev/null | grep "compliance_pct"

Scan Output Per Subscription

Each scan job reports: total resources, compliant count, non-compliant count, compliance percentage, and up to 20 detailed violations (resource ID + policy name). Results uploaded as artifacts and summarized in the workflow step summary.

Step 4: Self-Hosted Runner (Optional)

All 6 workflows use vars.RUNNER_LABEL to determine the runner. By default, workflows run on GitHub-hosted runners (ubuntu-latest). To use a private self-hosted runner:

StepAction
1Register a self-hosted runner on your GitHub repo or organization
2Go to Settings → Secrets and variables → Actions → Variables
3Create variable: Name = RUNNER_LABEL, Value = self-hosted (or your custom label, e.g., alz-runner)
# Set via GitHub CLI gh variable set RUNNER_LABEL --body "self-hosted"

How it works: The reusable deploy workflow receives the runner label as an input from the calling workflow. All other workflows read vars.RUNNER_LABEL directly, falling back to ubuntu-latest if not set. No YAML changes needed — one variable controls all 23 jobs across all 6 workflows.
