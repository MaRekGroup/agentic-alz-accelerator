Application Landing Zones are workload subscriptions placed under the **Landing Zones** management group. Each uses a profile that determines its networking, security, compliance, and monitoring posture.

They depend on the [Platform Landing Zones](/concepts/platform-landing-zones) being deployed first — specifically the Management (logging) and Connectivity (hub networking) subscriptions.

## Management Group Placement

```
Landing Zones
├── Corp          ← corp, sap profiles
├── Online        ← online profile
└── Sandbox       ← sandbox profile
```

## Profiles

### Online (internet-facing)

For internet-facing web applications, APIs, and e-commerce workloads protected by WAF.

| Attribute | Value |
|-----------|-------|
| Management Group | Online |
| Networking | Hub-spoke with WAF (Application Gateway) |
| DDoS Protection | ✅ Enabled |
| Private DNS Zones | Not required (public endpoints) |
| Policy Initiatives | Azure Security Benchmark, CIS Azure Foundations |
| Defender Plans | VMs, App Services, SQL, Storage, Key Vaults |
| Log Retention | 90 days |
| Microsoft Sentinel | ❌ Not required |
| Key Vault | ✅ Required |
| Use Case | Internet-facing web apps, APIs, e-commerce |

:::tip
Online landing zones route inbound traffic through Application Gateway with WAF v2 in Prevention mode. Outbound traffic is inspected by the hub Azure Firewall.
:::

### Corp (internal private)

For internal corporate workloads with private connectivity to on-premises via ExpressRoute or VPN.

| Attribute | Value |
|-----------|-------|
| Management Group | Corp |
| Networking | Hub-spoke with ExpressRoute/VPN peering |
| DDoS Protection | ✅ Enabled |
| DNS Servers | On-premises DNS |
| Private DNS Zones | SQL, Blob, Key Vault, and more |
| Policy Initiatives | Azure Security Benchmark, ISO 27001, NIST SP 800-53 |
| Custom Policies | `deny-public-ip`, `enforce-private-endpoints`, `deny-public-storage`, `require-encryption` |
| Defender Plans | VMs, SQL, Storage, Key Vaults, DNS, ARM |
| Log Retention | 365 days |
| Microsoft Sentinel | ✅ Required |

:::caution
Corp landing zones enforce **no public IP addresses** and **mandatory private endpoints** for all PaaS services via Azure Policy. Workloads that require public ingress should use the **Online** profile instead.
:::

---

### Sandbox (dev/test)

For developer experimentation and proof-of-concept workloads with relaxed policies and hard budget caps.

| Attribute | Value |
|-----------|-------|
| Management Group | Sandbox |
| Networking | Standalone VNet (no hub peering) |
| DDoS Protection | ❌ Not required |
| Policy Initiatives | Azure Security Benchmark (audit only) |
| Custom Policies | `allowed-vm-skus`, `budget-enforcement` |
| Defender Plans | VMs only |
| Log Retention | 30 days |
| Microsoft Sentinel | ❌ Not required |
| Budget | $500/month (hard cap via policy) |

:::note
Sandbox subscriptions have **no connectivity** to the hub network or on-premises. They are fully isolated. The $500/month budget is enforced by a `deny` policy that blocks resource creation when the budget is exceeded.
:::

---

### SAP (SAP workloads)

For SAP S/4HANA and related workloads requiring high-performance networking, proximity placement groups, and Azure NetApp Files.

| Attribute | Value |
|-----------|-------|
| Management Group | Corp |
| Networking | Hub-spoke + ExpressRoute + ANF subnet |
| DDoS Protection | ✅ Enabled |
| Proximity Placement Groups | ✅ Required for HANA performance |
| ANF Delegation | `Microsoft.NetApp/volumes` |
| Policy Initiatives | Azure Security Benchmark, SAP on Azure Best Practices |
| Accelerated Networking | ✅ Required on SAP app + DB subnets |
| Defender Plans | VMs, SQL, Storage, Key Vaults, DNS |
| Log Retention | 365 days |
| Microsoft Sentinel | ✅ Required |

:::caution
SAP landing zones require **Accelerated Networking** on all VM NICs in the app and database subnets. The ANF subnet must be delegated to `Microsoft.NetApp/volumes` and sized for the expected HANA data + log volumes.
:::

## Choosing a Profile

| Need | Profile | Key Differentiator |
|------|---------|-------------------|
| Public-facing web apps/APIs | **Online** | WAF + public endpoints allowed |
| Internal corporate workloads | **Corp** | Private endpoints only, on-prem connectivity |
| Developer experiments | **Sandbox** | Isolated, budget-capped, relaxed policies |
| SAP S/4HANA | **SAP** | ANF, proximity groups, accelerated networking |

## Deployment

Application Landing Zones are deployed via the `3-app-deploy.yml` GitHub Actions workflow. See the [Application Deployment guide](/guides/application-deployment) for step-by-step instructions.
