## Philosophy

**AI Orchestrates · Humans Decide · Azure Executes**

The accelerator uses specialized AI agents that collaborate through a structured
multi-step workflow. Each agent has a defined responsibility, produces specific
artifacts, and hands off to the next agent in the pipeline.

## System Architecture

```
┌─────────────────────────────────────────────────────┐
│                  🧠 Conductor                        │
│              (Master Orchestrator)                    │
│  Routes workflow · Enforces gates · Tracks state     │
└──────────────────────┬──────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
   ┌────▼────┐   ┌────▼────┐   ┌────▼────┐
   │ Per-LZ  │   │ Estate  │   │ Day-2   │
   │ Agents  │   │ Agents  │   │ Agents  │
   │ (0-7)   │   │ (Gov)   │   │ (8-9)   │
   └─────────┘   └─────────┘   └─────────┘
```

### Per-LZ Agents (Steps 0-7)

Execute the full APEX workflow for a single landing zone — from assessment
(brownfield) or requirements (greenfield) through deployment and documentation.

### Estate Agents

Operate across all landing zones for cross-cutting governance, policy
enforcement, and RBAC management.

### Day-2 Agents (Steps 8-9)

Continuous operations — compliance monitoring, drift detection, and
auto-remediation with rollback capability.

## Key Principles

1. **Human-in-the-Loop** — 6 approval gates where humans review and approve
2. **AVM-First** — Azure Verified Modules for standardization
3. **Security by Default** — 6 non-negotiable security rules enforced everywhere
4. **Cost Governance** — Budget alerts mandatory on every deployment
5. **No Local Azure CLI** — All Azure operations through GitHub Actions
6. **Prefix-Based Naming** — Everything parameterized for multi-tenant reuse

## Artifact Flow

Each workflow step produces a numbered artifact that feeds into the next step:

```
00-assessment-*.{md,json,mmd} (brownfield only)
    → 01-requirements.md → 02-architecture-assessment.md → 03-design-*.md
    → 04-governance-constraints.md → 04-implementation-plan.md
    → infra/{bicep,terraform}/ → 06-deployment-summary.md → 07-*.md
```

All artifacts are stored in `agent-output/{customer}/` with estate-level
state tracked in `agent-output/{customer}/00-estate-state.json`. Brownfield assessment
artifacts are stored separately in `agent-output/{customer}/assessment/<scope>/`.

## Architecture Layers

The system has 6 layers from user interface to Azure resources:

| Layer | Components |
|-------|------------|
| **Presentation** | VS Code Copilot Chat, CLI, Azure Portal |
| **Orchestrator** | Conductor (DAG Workflow), Approval Gates, Session State, Artifact Handoffs |
| **Agent Layer** | � Assessor, �📜 Scribe, 🏛️ Oracle, 🎨 Artisan, 🛡️ Warden, 📐 Strategist, ⚒️ Forge, 🚀 Envoy, 📚 Chronicler, 🔭 Sentinel, 🔧 Mender, ⚔️ Challenger |
| **MCP Servers** | Azure Pricing (cost estimation), Azure Platform (Resource Graph, Policy, Deploy, Monitor), Draw.io (diagrams) |

| **Azure** | Management Groups, Subscriptions, Networking, Identity, Security, Governance |

## MCP Server Details

3 Model Context Protocol servers connect agents to live Azure data:

| MCP Server | Key Tools | Used By |
|-----------|-----------|----------|
| **Azure Pricing** | Cost estimation, region comparison, RI pricing, budget calculation | Oracle, Strategist, Chronicler |
| **Azure Platform** | Resource Graph queries & inventory, Azure Policy discovery & compliance, Bicep/TF deploy & what-if, Defender secure score, Activity log & drift detection | All agents |
| **Draw.io Diagrams** | Architecture diagram generation, shape search, diagram export, Azure icon library | Artisan, Chronicler |

:::tip
MCP servers are configured in `mcp/mcp-config.json`. The Azure Platform server consolidates 27 tools into a single server.
:::
